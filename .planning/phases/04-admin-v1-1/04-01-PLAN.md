# Phase 4 — Plan 01: Admin 外壳 + 密码守卫 + 条目浏览器

## Goal

维护者访问 `/admin`，经前端密码门禁进入编辑工作台；工作台经 `getAllAnime()`/`getAllTools()` 加载并展示全部追番与工具条目，Tab 切换两种条目类型，整卡点击进入未来编辑（Phase 5），删除图标按钮二次确认后视觉隐藏 + 写 localStorage。覆盖 EDIT-01、EDIT-02。

## Verification

1. `npm run build` exit 0（schema 校验通过、define:vars 序列化正常）
2. `npm run preview` 后浏览器人工走查：错误密码留页 + 行内提示；正确密码进入工作台；Tab 切换；删除确认 + 刷新持久化
3. 查看 `/admin` 源码：`<meta name="robots" content="noindex, nofollow">` 存在；密码视图默认可见（`#pw-view` 无 hidden）、工作台默认 hidden（`#dashboard` 含 hidden class）

## Context / Patterns

C-01 服务端渲染完整卡片网格，客户端仅持极简数据副本 `{ id, type, slug, title }` 经 `define:vars` 注入——不传完整 Content Entry（含 render 函数/image 对象/Date 不可序列化，否则客户端拿到 undefined/`[object Object]`）。

C-02 默认 HTML 状态 = 未登录态（`#pw-view` 可见、`#dashboard` 含 hidden class）。客户端脚本是「增强」路径：已登录用户进入工作台后切走密码视图，不会产生逆向 FOUC。

C-03 删除状态刷新后不丢：客户端初始化读 `chuxi_admin_deleted` 集合，对匹配卡片加 `hidden` class（视觉叠加在服务端渲染上）。Phase 7 导出基于同一 localStorage key 保证导出的 JSON 不含已删条目。

C-04 整卡点击与删除按钮分离：卡片的删除按钮用绝对定位兄弟 `<button type="button" data-delete={id} data-type={type}>` 放置在 `bottom-2 right-2`，避免与 AnimeCard StatusPill (`right-2 top-2`) 和 EntryCard 外链按钮 (`right-3 top-3`) 重叠。事件委托在 `#dashboard` 上监听点击（`e.target.closest('[data-delete]')`），不冒泡进 `<a>`。

C-05 本章不引入任何新依赖、新组件文件、新 lib 文件——零安装。
